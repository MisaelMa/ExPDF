3 Lazy indirect-object resolver with Map-based cache.
4
5 Per the design (`sdd/pdf-reader-core/design` § 5 "Lazy Resolution Contract"):
6
7 - **Cache**: a plain `Map` on `%Pdf.Reader.Document{}.cache`. No GenServer.
8 Key: `{obj_num, gen_num}`. Value: the resolved Elixir term.
9 - **Signature**: `resolve(doc, {:ref, n, g}) :: {:ok, value, doc} | {:error, reason}`.
10 The returned `doc` carries the updated cache. The caller threads `doc` forward
11 for cache benefit; dropping the updated doc still yields correct results on
12 the next call (re-parse, same value — the binary is immutable).
13 - **Idempotent**: calling `resolve/2` twice on the same ref with the same doc
14 returns the same value. The cache is a hint, not state.
15
16 ## Resolution paths
17
18 1. **Cache hit**: `Map.get(doc.cache, {n, g})` → immediate return.
19 2. **In-use (classic)**: look up `{n, g}` in `doc.xref` → `{:in_use, offset, _gen}`.
20 Slice `binary_part(doc.binary, offset, ...)`, run `Parser.parse_object/1`.
21 3. **Compressed (ObjStm)**: look up `{n, g}` → `{:compressed, objstm_n, index}`.
22 Recursively `resolve(doc, {:ref, objstm_n, 0})`, decode filters, then
23 `ObjectStream.fetch/3`.
24 4. **Free / absent**: `{:error, {:unresolved_ref, {n, g}}}`.
25
26 ## Ref chasing
27
28 `resolve/2` does **NOT** automatically follow nested refs. If a resolved value
29 is itself `{:ref, _, _}`, the caller decides whether to chase it. This avoids
30 infinite loops on circular references and keeps the interface predictable.
31 """
